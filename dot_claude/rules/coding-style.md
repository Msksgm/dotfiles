# コーディングスタイル

## 不変性（CRITICAL）

オブジェクトを直接変更せず、常に新しい値を返すこと:

```
// 擬似コード
// bad: 元のオブジェクトをその場で変更する
modify(original, field, value) → original が変わる

// good: 変更を反映した新しいコピーを返す
update(original, field, value) → 変更後の新しいオブジェクトを返す
```

理由: 不変なデータは隠れた副作用を防ぎ、デバッグを容易にし、安全な並行処理を可能にする。

## コア原則

### KISS（シンプルに保つ）

- 実際に動く最もシンプルな解を選ぶ
- 早すぎる最適化をしない
- 賢さよりも読みやすさを優先する

### DRY（繰り返しを避ける）

- 繰り返されるロジックは共有関数やユーティリティに切り出す
- コピーペーストによる実装のズレを避ける
- 繰り返しが実際に存在するときに抽象化する（投機的な共通化はしない）

### YAGNI（必要になるまで作らない）

- まだ必要でない機能や抽象化は作らない
- 投機的な汎化を避ける
- シンプルに始め、実際に必要になってからリファクタリングする

### SOLID 原則

- **単一責任（S）**: クラス・モジュールは1つの責務に集中させる
- **開放/閉鎖（O）**: 拡張に対してオープン、変更に対してクローズ
- **リスコフ置換（L）**: サブタイプはスーパータイプと置換可能にする
- **インターフェース分離（I）**: クライアントが使わないメソッドに依存させない。ただし過度なインターフェースの使用は避けること
- **依存性逆転（D）**: 具体ではなく抽象に依存する

## ファイル構成

少数の大きなファイルより多数の小さなファイルを優先する:
- 高凝集・低結合
- 200〜400行が標準、最大800行
- 大きなモジュールからはユーティリティを切り出す
- 種別（controllers/models 等）ではなく、機能/ドメインで整理する

## エラーハンドリング

すべてのレイヤーで明示的にエラーを処理すること:
- エラーを握りつぶさない。最低限ログに出力すること
- エラーを型として表現し、呼び手側で明示的に扱えるようにすること
- UI 向けコードでは、ユーザーが理解できるエラーメッセージを提供する
- サーバー側では、詳細なエラーコンテキストをログに残す
- `raise` された例外が呼び手側でどう扱われているかに注意すること。何もハンドリングされていない場合は質問すること

## 入力バリデーション

システム境界では必ずバリデーションを行うこと:
- ユーザー入力はすべて処理前に検証する
- 利用可能な場合はスキーマベースのバリデーションを使う
- 明確なエラーメッセージとともにフェイルファストする
- 外部データ（APIレスポンス、ユーザー入力、ファイル内容）は絶対に信頼しない

## 避けるべきコードの問題

### 深いネスト

ロジックが積み重なってきたら、ネストした条件分岐より早期リターンやガード節を使うこと。
`private` 関数から `private` 関数をネストして呼び出すことはしないこと。ネストが2回以上になるのは別のクラスに切り出すヒント。

### マジックナンバー・マジックストリング

意味のある閾値・遅延・制限には名前付き定数を使うこと。

### 長い関数

大きな関数は責務が明確な小さな単位に分割すること（目安: 50行以下）。
ただし無理に切り出す必要はなく、処理を追いにくくなる過度な分割は避けること。

## コード品質チェックリスト

作業完了とする前に確認すること:
- [ ] コードが読みやすく、適切に命名されている（詳細は `@rules/coding-general.md` 参照）
- [ ] 関数が小さい（50行以下）
- [ ] ファイルが集中している（800行以下）
- [ ] 深いネストがない（4レベル以下）
- [ ] エラーが適切に処理されている
- [ ] ハードコードされた値がない（定数または設定を使用）
- [ ] 不変パターンが使われている（直接変更なし）
