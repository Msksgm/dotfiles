# コードレビュー基準

## 目的

コードレビューは、コードがマージされる前に品質・セキュリティ・保守性を担保するために行う。

## レビューすべきタイミング

**必須レビューのトリガー:**
- コードを書いた・変更した後
- 共有ブランチへのコミット前
- セキュリティに関わるコードを変更したとき（認証・決済・ユーザーデータ）
- アーキテクチャ上の変更を加えたとき
- プルリクエストのマージ前

**レビュー依頼前の前提条件:**
- すべての自動チェック（CI/CD）が通過していること
- マージコンフリクトが解消されていること
- ブランチがターゲットブランチと同期されていること

## レビューチェックリスト

コード完了とする前に確認すること:
- [ ] コードが読みやすく適切に命名されている
- [ ] 関数が集中している（50行以下）
- [ ] ファイルが凝集している（800行以下）
- [ ] 深いネストがない（4レベル以下）
- [ ] エラーが明示的に処理されている
- [ ] シークレットや認証情報がハードコードされていない
- [ ] デバッグ出力が残っていない（`console.log` や `print` 等）
- [ ] 新機能にテストが存在する
- [ ] テストカバレッジが80%の最低基準を満たしている

## セキュリティレビューのトリガー

以下の変更を加えた場合は**必ず停止し、code-reviewer agent でセキュリティ観点のレビューを行うこと:**
- 認証・認可コード
- ユーザー入力の処理
- データベースクエリ
- ファイルシステム操作
- 外部 API 呼び出し
- 暗号処理
- 決済・金融コード

## 深刻度レベル

| レベル | 意味 | 対応 |
|-------|------|------|
| CRITICAL | セキュリティ脆弱性またはデータ損失リスク | **BLOCK** — マージ前に必ず修正 |
| HIGH | バグまたは重大な品質問題 | **WARN** — マージ前に修正すべき |
| MEDIUM | 保守性の懸念 | **INFO** — 修正を検討する |
| LOW | スタイルや軽微な提案 | **NOTE** — 任意 |

## Agent の使い方

当環境で利用可能なレビュー agent:

| Agent | 目的 |
|-------|------|
| **code-reviewer** | コード品質・パターン・ベストプラクティス全般、セキュリティ脆弱性の確認 |
| **qa-reviewer** | テスト網羅性・不足テストケースの特定 |
| **architect** | アーキテクチャ変更の妥当性確認 |
| **pm-reviewer** | 仕様・プランの抜け漏れ検出 |

## レビューワークフロー

```
1. git diff で変更内容を把握する
2. セキュリティチェックリストを先に確認する
3. コード品質チェックリストを確認する
4. 該当するテストを実行する
5. カバレッジが 80% 以上であることを確認する
6. 詳細レビューのために適切な agent を使う
```

## よくある問題

### セキュリティ

- ハードコードされた認証情報（APIキー・パスワード・トークン）
- SQLインジェクション（クエリへの文字列連結）
- XSS脆弱性（サニタイズされていないユーザー入力）
- パストラバーサル（サニタイズされていないファイルパス）
- CSRF保護の欠如
- 認証バイパス

### コード品質

- 大きすぎる関数（50行超） — より小さな単位に分割する
- 大きすぎるファイル（800行超） — モジュールを抽出する
- 深いネスト（4レベル超） — 早期リターンを使う
- エラー処理の欠如 — 明示的に処理する
- ミュータブルな操作 — 不変パターンを好む
- テスト不足 — テストカバレッジを追加する

### パフォーマンス

- N+1クエリ — JOIN またはバッチ処理を使う
- ページネーション未実装 — クエリに LIMIT を追加する
- 制限のないクエリ — 制約を追加する
- キャッシュ未実装 — コストの高い操作をキャッシュする

## 承認基準

- **承認**: CRITICAL・HIGH の問題がない
- **警告**: HIGH の問題のみ（注意してマージ可）
- **ブロック**: CRITICAL の問題あり
