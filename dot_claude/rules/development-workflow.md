# 開発ワークフロー

> このファイルは `@rules/git-workflow.md` を拡張し、git 操作前の機能開発プロセス全体を定義する。

機能実装ワークフローは、調査・計画・TDD・コードレビュー・コミットという開発パイプラインを規定する。

## 機能実装ワークフロー

0. **調査・再利用** _（新規実装前の必須ステップ）_
   - **既存実装の探索を最初に行う**: 新たに書く前に、解決済みの実装・テンプレート・パターンをまず探すこと
   - **ライブラリ・パッケージの活用**: ユーティリティコードを手書きする前に、既存パッケージやライブラリを調査する。実績のあるライブラリを手製ソリューションより優先する
   - **一次ソースドキュメントを確認する**: API の挙動・パッケージ利用方法・バージョン固有の詳細は、実装前に公式ドキュメントで確認する
   - **適応可能な実装を探す**: 問題の 80%以上を解決できるオープンソースプロジェクトがあれば、フォーク・移植・ラップを検討する
   - 実績ある手法の採用・移植を、ゼロからの新規実装より優先すること

1. **計画を立てる**
   - **architect** agent を使って実装計画を作成する
   - コーディング前に計画ドキュメントを生成する（アーキテクチャ・依存関係・タスクリスト等）
   - 依存関係とリスクを洗い出す
   - フェーズに分解する

2. **TDD アプローチ**
   - **qa-reviewer** agent を活用する
   - テストを先に書く（RED）
   - テストが通るように実装する（GREEN）
   - リファクタリングする（IMPROVE）
   - カバレッジが 80%以上であることを確認する

3. **コードレビュー**
   - コードを書いた直後に **code-reviewer** agent を使う
   - CRITICAL・HIGH の問題を対処する
   - MEDIUM の問題はできる限り修正する

4. **コミット・プッシュ**
   - 詳細なコミットメッセージを書く
   - conventional commits フォーマットに従う
   - コミットメッセージフォーマットと PR プロセスは `@rules/git-workflow.md` を参照すること

5. **レビュー前チェック**
   - すべての自動チェック（CI/CD）が通過していることを確認する
   - マージコンフリクトをすべて解消する
   - ブランチがターゲットブランチと同期されていることを確認する
   - これらのチェックが完了してからレビューを依頼すること
