# テスト要件

## テストカバレッジの最低基準: 80%

テスト種別（すべて必須）:
1. **ユニットテスト** — 個々の関数・ユーティリティ・コンポーネント
2. **インテグレーションテスト** — APIエンドポイント・データベース操作
3. **E2Eテスト** — 重要なユーザーフロー（言語・フレームワークに応じて適切なツールを選択）

## テスト駆動開発（TDD）

必須ワークフロー:
1. テストを先に書く（RED）
2. テストを実行する — **失敗**することを確認する
3. 最小限の実装を書く（GREEN）
4. テストを実行する — **通過**することを確認する
5. リファクタリングする（IMPROVE）
6. カバレッジを確認する（80%以上）

## テスト失敗時のトラブルシューティング

1. **qa-reviewer** agent を使ってテストの問題を分析する
2. テストの独立性を確認する（テスト間で状態が共有されていないか）
3. モックが正しく設定されているか確認する
4. テストを修正するのではなく実装を修正する（ただしテスト自体が間違っている場合を除く）

## Agent サポート

- **qa-reviewer** — 新機能や機能改修において積極的に利用し、テストファーストを徹底する

## テスト構造（AAA パターン）

Arrange-Act-Assert（準備・実行・検証）の構造でテストを書くこと:

```
// 例（言語に依らない擬似コード）
test('コサイン類似度が正しく計算される') {
  // Arrange: テストデータを準備
  vector1 = [1, 0, 0]
  vector2 = [0, 1, 0]

  // Act: テスト対象を実行
  similarity = calculateCosineSimilarity(vector1, vector2)

  // Assert: 結果を検証
  expect(similarity).toBe(0)
}
```

## テスト命名規則

テストされる振る舞いが伝わるような説明的な名前をつけること:

```
// good
test('クエリに一致するマーケットがない場合は空配列を返す')
test('APIキーが欠落している場合はエラーをスローする')
test('Redisが利用できない場合はサブストリング検索にフォールバックする')
```
